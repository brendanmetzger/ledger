First-class Functions

From a linguistic perspective, functional programming may seem to be nothing more than writing a program with functions, but this not accurate—all programs are written with functions in a simple term. Functional programming hellbent on austerity. Conversely, in imperative programming, we solve problems as straightforward as possible; sequentially through loops, conditions and function calls. Functional programs solve problems only with functions, largely out of sequence and order. While we will use the imperative style to solve our problems I’m introducing the contrast between them because JavaScript can be a be clever and applies many functional concepts; such cleverness can be frustrating depending on your level of experience.

A first-class function is a fundamental part of the language, like our types (number, string, boolean, etc). Compare the two greeting variables in the following example and note that we can stuff a function into a variable.

This no doubt looks a bit odd, but the important thing to notice is that both examples produce a similar outcome: they assign a string with a typical english greeting to a variable, but they take a very different approach. Just like a number indicates quantitative property, such as measurement, function represents an action to take place. However, that action does not take place until you call the function with the () operator. In this example, I'm doing that immediately after I declare the function body (which is some of the cleverness I mentioned earlier).

A function starts as an idea of something to be done written in the function definition. Once defined, it can be executed at any time to produce value. Pause there: it produces a value by returning something; numbers, strings, booleans, arrays, objects, even other functions! This is the first-class shenanigans I'm talking about: functions can be tossed around like any other type.

Some Examples: A loaf of bread

Before you begin making bread, you absolutely, fundamentally, completely know what a variable is and how to assign a value to it. Further, you understand strings, numbers and booleans. At this point, you have a decent idea of what an object is, and its sub-type, the array. You’re quite familiar with the syntax required: quotes, square and curly brackets, and parentheses. If any of this is a mystery, rewind!

At the most abstract level, we need an interface that produces a loaf of bread.

We first have to say what producing bread is before we apply it. We send along any expected arguments as parameters during application—in this case, our declaration expects ingredients as an argument, and when applied, we send along an array as a parameter. Do not go further until you understand declaring vs. applying functions and arguments vs. parameters..
But producing bread is probably better described as composite of several actions.

We have ingredients, but they have to be mixed, and after mixed, they have to be baked. Let’s assume we have declared functions called mix and bake that can take care of that for us. Note: what is returned when mix and bake are applied? Never fail to consider input vs. output.
But producing bread, at least in most situations, involves a person.

The previous function is great, but thinking a bit deeper, it assumes that all bread is made the same way. Any bread fan knows that two different bakers, even using the same ingredients, can make two entirely different loaves. Let’s create a notion of a baker.
First-Class Lesson 1: declared functions can return un-applied functions.

A Baker is really a person who bakes, so let’s just consider that one facet of an entire human. Here I declare two simplified people, both capable of baking. They're not baking 100% of the time; remember, they will only bake when called.

In our highlighly simplified version, we are just going to assume that when either Sue or Lou is applied, they return a Baker. Do not forget that a Baker is a function itself: look very closely at how they return a Baker, uncalled. That is, think of it as returning the occupation of a Baker, not someone actually baking..

Do not let this slip past you.
First-Class Lesson 2: Functions can have un-applied functions as parameters.

There was a notion of this above in the Baker declaration when we specified and used the performAction argument—an extremely important concept, so important, that it has been given it's own descriptive name: a callback. Our Baker function wants to be given a task (function) and some material as parameters, it will then apply the task with material on its own. Don’t miss the following point, mix and bake are only declared—they will be executed by something or someone else.

Dates and Times

Timing in javascript kind of falls into two camps: the time you can control, and the time you can’t. The latter refers to asynchronous aspects of the language—aspects dealt with by using event listeners and things like intervals and timeouts. The former, time we do not control, is what we will be concerned with here.

The Date Object

The JavaScript Date object has always been a tad clunky in my opinion (at least compared to other languages), but should you need it—and you probably will, there are a few tricks to remember, and the first oddity is that it must be called as a constructor—this means you must use new like below.

Deciding when and when not to use new confuses even those with experience, so some general advice is to avoid new unless you have to explicitly use it, like with the Date object.

The constructor function takes a variety of arguments, which you can read about in the documentation, but if you carefully write your dates, you can pass a string. Given '1929-10-05 08:10:00'; we read that as October 5, 2012, 8:10am… but am or pm? When you write you are using the 24 hour clock so the string will be parsed accurately.

When constructing a date from a string, you can pass the string directly to the constructor if it is formatted properly—there are standards committees mandate acceptable formats—those are available in the documentation. A few static Date methods to be aware of: Date.parse(timestring) will return the number of milliseconds since January 1, 1970. This is immediately useful, say if you want to calculate the difference between two dates: Date.parse('some date') - Date.parse('some other day');, then you could convert from milliseconds into hours/min, etc, with some arithmetic. Another nice method is Date.now(), which gives you the number of milliseconds at the current moment.

Naming Conventions

This is both the easiest thing to learn, also the easiest to skip, so it is worth revisiting often. As you get more familiar with writing source code, your efforts transcend from getting it to work into getting it to work in a way someone else can understand.

Naming Conventions are a set of rules for choosing the character sequence to be used for identifiers, be it classes, IDs or variables and functions in JavaScript. In html distinguishing between upper and lowercase is not extremely important for correct page rendering, but tag names should be in lowercase. In css and JavaScript, upper and lowercase words are entirely different. Likewise, whitespace is another obstacle—css classes and IDs as well as JavaScript variables cannot start with a number, and cannot contain any whitespace. In many ways, learning to program is simply learning to become extremely careful about naming things and typing those names.

This is something you generally learn over time, but standard cleverness/carefulness applies now. You can prefix and suffix IDs with common words to keep them unique but class-like. You can develop some taxonomy for css for common usage. In JavaScript—and really all programming—developers take this to another level. Not only is important to be consistent in your own work, there are many agreed upon conventions and techniques. Browse these JavaScript conventions and try to begin implementing some of the techniques.

While not particularly noteworthy, consistency is something that takes a lot of practice and effort. Try to keep your variable names, class names and any other name in a fashion where you (and others!) can make sense of what your code is doing.

Snake case is low_like_a_snake

Snake case is a way of having multi-word strings by replacing white space with an underscore. In CSS, it’s common to use a hyphen (-) as well.

Camel Cased is bumpyLikeAcamel

Camel Case is when you separate multi-word strings by replacing white space with the adjoining word converted to uppercase.

Namespace and Scope

As programs grow, it becomes harder (and it is generally unwise) to add more and more complicated variable and class names. To combat this obvious setback, a highly organized individual will employ namespacing techniques. One way think of namespacing is imagining that every car in America had an federal license plate instead of by state. Think about what that would do; only one person would be able to register a plate with POOPS. Namespaced to each individual state, and all of a sudden each state can have the full spectrum of options. Now imagine each county had their own plate, and so on and so forth.

Just like adding folders to your computer, this organization principle is essentially namespacing. With CSS, this can be accomplished with classes, i.e., .outer .inner would only apply inner if a parent has an outer, thus giving .inner different meanings depending on context. In JavaScript, this is accomplished with objects, i.e., var bob = { age: '...'}, or through clever uses of functions and scope. Say you have two object variables, bob and sally, each representing the same type of object; bob.age and sally.age have namespaced the age property.

From First-Class to Callbacks

If you've ever heard the term callback and thought it sounded lofty, then rejoice, if you’ve been following along, then you already know more than you think.

Well known at this point is that functions are first declared and then applied. Sometimes we do the declaring, sometimes the language has defined functions for us, like Math.cos();. Likewise, sometimes we take charge of applying a function directly, sometimes we declare a function and never actually apply it ourselves. A function can be applied immediately, or applied later on by another function if it is passed in an uncalled state.

It is the latter method that is officially known as a callback—nothing more than an undeclared function passed as a parameter to another function. You must read and understand the following notion: saying something like a function passed in an uncalled state is rather difficult, so it was wrapped into its very own word. That's right, we say callback when talking about a previously declared function being passed around in its uncalled state. Commit this to memory—from here on out, when you hear callback, you should know the precise meaning.

A function given a callback determines when and how the callback is applied; this is how JavaScript does business. It is powerful and it’s this very concept that allows you to do event-based programming. If you are an interactive designer, this is a way you should start framing design problems/solutions.

So, consider this: if a mouse clicks on a webpage but a program doesn't hear it, does it make a sound?.

Now that we know about first-class functions and callbacks, we can actually consider events, and there are tons of events that we can take advantage of.

EventTarget

Typical objects are any HTML Element or window object, but there are hundreds of events that can be applied to many other objects.

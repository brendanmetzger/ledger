From First-Class to Callbacks

If you've ever heard the term callback and thought it sounded lofty, then rejoice, if you’ve been following along, then you already know more than you think.

Well known at this point is that functions are first declared and then applied. Sometimes we do the declaring, sometimes the language has defined functions for us, like Math.cos(). Likewise, sometimes we take charge of applying a function directly, sometimes we declare a function but we don’t apply it ourselves; functions can be applied immediately or by another function—pending we send it as a parameter in an uncalled state.

It is the latter method that is officially known as a callback; simply a declared function passed as a parameter to another function. You must read and understand the following notion: saying something like a function declaration passed in an uncalled state is mouthful of a concept, so it been granted it’s own word as a concept. That's right, we say callback when talking about a previously declared function being passed around in its uncalled state. Commit this to memory—from here on out, when you hear callback, you will know the precise meaning.

A function given a callback determines when and how the callback is applied; this is how JavaScript does business. It is powerful and it’s this very concept that allows you to do event-based programming. If you are an interactive designer, this is a way you can/should start framing all sornts of design problems/solutions.

Events

If a tree falls in a forest and no one is around to hear it, does it make a sound?

Events

In this classic thought experiment we are engaged to think about all of the events that can occur without our knowing, and then get metaphysical about whether or not the event ever occured. Clearly, this seemingly simple question has been the substance of deep philosophical thought for centuries, so I think it worth five minutes of our time. So, consider this: if a mouse clicks on a webpage but a program doesn't hear it, does it make a sound?.

Now that we know about first-class functions and callbacks, we can actually consider events, and there are tons of events that we can take advantage of. There are several ways to capture events, but the best, and preferred way, is through a built-in function of the JavaScript DOM called addEventListener. Talking about what to when an event occurs always starts with the premise that you must be able to perceive an event, so it should be easy to remember add • event • listener as a concept. You have all the objects on a page, how do they know when something happens? You tell them to listen.

There is another way to listen for events by setting attribute on an element (either in HTML or JavaScript)—you will no doubt see this technique often. While convenient and easy, I rarely (almost never) do this for valid technical reasons, but more importantly, it violates a design principle called separation of concerns. We have three facets that control interactive applications; semantics and structure (HTML), view and design (CSS), and behavior (JavaScript)—best to keep them separate.

Now considering the process of an event occurring, remember that it is emmitted from a source, be it an Element or the window object, etc., and you must apply a listener directly to that source in order to hear it. It might occur to you that Elements have children and parents; begin the mental process of thinking about what that might mean for how the event propogates or bubbles.

Usage and Documentation

Function Signature

A function signature specifies the general information about a function like the name, number and type arguments and what it returns. This information can usually be found in the documentation of a language or framework—I doubt anyone flat out reads an entire manual, but knowing how to read it and knowing where to find it is mandatory.

Definition

void1 addEventListener(string event2, function callback3, [boolean capture4]);
The name of the function is underlined in black text
1 The return value of the function - in this case, void indicates that the function returns nothing
2 The first argument expects a string, in this case, the type of event ('click', 'scroll', etc.)
3 The next argument is a callback which must be a type function (yet-to-be called).
4 Capture—this is an obscure point, the fact that it in brackets indicates it is optional.
The Callback Signature
Whenever you write a callback, you should define an argument—you can call it whatever you like, I generally name it evt
When an event is emitted, and you’re listener hears it, it will apply your callback method. When applied, it passes along the Event object as a parameter.
Event
The Event object is passed along as a parameter to your callback whenever it occurs. It has dozens of properties
You can make custom events (not very common)
EventTarget
The EventTarget interface is implemented by any object that can receive events and have listeners applied to them
Typical objects are any HTML Element or window object, but there are hundreds of events that can be applied to many other objects.

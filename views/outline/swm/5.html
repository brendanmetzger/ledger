Pseudo Classes

A class is added to an element to apply a style. In theory, a developer could create a perfect stylesheet with well defined classes, and then apply those classes to HTML later on, as necessary. With practice, you often write a stylesheet to set a baseline for yourself that fits a pattern of your development techniques, such as reseting the font size and margin of certain elements before you write any markup whatsoever. It is wise to consider what sort of classes you should always have on hand (like .container I mentioned in our intro to floats) that facilitate certain patterns you are sure to encounter on every project. On that note, let us consider some patterns.

For example, say we are developing a typographic style guide, and we would like the first paragraph of every article to be 150% larger than the following paragraphs. Even though our style guide is only conceptual at this point, we know that we should be thinking about how to deal with first paragraphs. We could make a class for that—say .enlarge that we apply to a <p> and that would work well for most purposes.

While that is a decent solution, and the generic nature of .enlarge is admirable, we didn't really address our problem as well as we could, and we are reliant on actually writing html to apply that class. At this point, we may notice the existential state that if there is at least one paragraph in an article, then some paragraph must be the first. Does this present us with an opportunity? The answer is yes, indeed there is an opportunity, and we can use pseudo classes.

The adjacent list notes many of the pseudo classes available. Some like :hover are hopefully quite familiar. Some like :focus and :enabled are only for form elements, which is more advanced fodder. My advice is to become friendly with the ordering pseudos, which are those containing first, last and nth.

Responsive Design

Responsive web design is a technique used to modify layout, typography and other design considerations in response to the size and type of device used by the viewer. It can be elusive and frustrating, so some philosophy is necessary.

Mobile First

Mobile-first design is an important concept I suggest you practice. Essentially, it means that you design something for the smallest screen you are supporting, and then you scale up to accommodate bigger screens and the more space you have available. The fact of the matter is that your mobile-first design is always very simple: it is only one column, type is generally designed to be read about 2 feet from a user’s face and interactive elements need to accommodate a fingertip.

While this seems like an obviously good choice, in practice, few designers (or clients) are patient enough for this process, for a couple of reasons. Often the mobile design is done after the desktop because then a client can approve and adjust the look and feel. In addition, most designers don't work in an interactive design medium (like a text-editor/browser) so they are not very aware of how responsive design even works.

From experience, these caveats delay a project significantly and it's painful and awkward to go from full-scale design to small, but relatively easy to go small to big. In other words, put on your skinny jeans before your holiday meal, not after. So if you want to get the knack of mobile development, always start mobile first.

Media Queries

Media Queries are pretty simple—if you have the gist of CSS, the won’t give you any trouble. Essentially, imagine you can divide your styles up by certain criteria: if the screen is too small, load this stylesheet, and likewise, you can target device types, landscape vs. portrait, etc. Say you asked yourself, I want some really big text, but only on a really big screen.. Media Queries are where you begin, and once you get your site schematics figured out and media in place they can offer immediate solutions to problems with text size and layout problems. The W3Schools reference is also a good starting point.

Notes:

There is one more technical item you should be aware of. Be sure to investigate mobile-appropriate meta tags to make sure your css is applied correctly. Remember, meta tags are used as instructions—they are invisible to users, but machines use that information to make other decisions.

Responsive design is a lot of philosophy and a little bit of technique. You've seen me employ all of the techniques you will ever need at this point: relative and viewport units, and now, media queries. From here, it becomes mostly strategy and having a good plan when you design, which we will work together on, but realistically takes years to appreciate.

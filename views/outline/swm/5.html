Responsive Design

Media Queries

Media Queries are pretty simple—if you have the gist of CSS, the won’t give you any trouble. Essentially, imagine you can divide your styles up by certain criteria; max/min screen sizes, device types, landscape vs. portrait, etc. Say you asked yourself, I want some really big text, but only on a really big screen.. Media Queries are where you begin, and once you get your site schematics figured out and media in place they can offer immediate solutions to problems with text size and layout problems. The W3Schools reference is also a good starting point.

New Units

When considering units, now would be a great time to start finding ways to implement vh and vw, which is your screen’s height and width broken down into 1% increments; ie. 10vh is 10% of the viewport height. Here is an intro to viewport sized typography

Mobile First

Mobile-first design is a concept that pairs well with media-queries and is quite important for designers and developers to understand and apply. Mobile-first basically means that you design something for the smallest screen, and then you scale up to accommodate bigger screens and the more space you have available. This can be difficult to do in practice, but in general, it helps to structure your media queries this way—essentially, use min-width in favor of max-width.

Be sure to investigate mobile-appropriate meta tags to make sure your css is applied correctly.

Try adjusting your screen size and ratio. The media query is set up to acknowledge a thin aspect-ratio—screen two times wider than it is tall—which will cause the background color to turn green. Any screen less than 1200px wide will adjust styles as well.

The Document Object Model

In the beginning all the technologies we are using probably seem very independent from one another as you gain experience you will see that they blur together in consistent ways. Let’s start with the things we know a little bit about.

The nature of the structural relationship of HTML is going to play a key role in how I talk about the Document Object Model. Parents and children, siblings, ancestors and descendants describe relationships, they inform the ruleset that describes how the interact, they are consequential! Note, I have been using those terms since the very start of the semester.

Document (Markup Languages)

Markup languages take text and media and convert them into structured data. We are becoming familiar with html, but html is really just the simple-minded cousin of the affluent and worldly xml, or eXtensible markup language. Boiled down very simply, html is nothing more than a subset of xml—that funny looking bit at the top called <!DOCTYPE html> defines which tags we can use and how to use them. You could write your own doctype and use any tags you want; people do this with xml allowing this very simple set of formatting rules. These formatting rules allow us to write plain-text documents that are the starting point for converting data into objects.

Object

The term object is a special word in the field of mathematics and computer science, representing various structures that can be described as having some distinct quality. When referring to the the document object model we are talking about the magic that happens after our plain old text—that has been carefully marked-up with / and < and > symbols—has been parsed by an algorithm to mean something, as a bunch of objects that are distinct, meaningful things. What once was a simple <h1> in text has been turned into an object that we can now communicate with. Hey, <h1>, I’m going to set your text color to blue. Tell me about your family... who’s your parent... any siblings? I’m going to move you somewhere else, oh and when someone clicks on you, let me know, ok?. An object, just like a toaster, has properties that describe what it is and how it works, and it has methods that allow it to perform certain tasks. Objects have properties and methods that we can communicate with, and this communication is known as programming.

Model (Tree Structures)

All together, The Document Object Model, or dom, is a standardized, cross-platform and language-independent convention for representing and interacting with objects in html, xhtml and xml documents. The objects, or nodes of every document are organized in a tree structure. Objects in the dom tree may be styled manipulated with other languages (wink/nod CSS, JavaScript).

You can visualize tree structures because they are represented by a branch of mathematics known as graph theory, which combines the rigor of math with the real-world pleasure of figuring things with your eyeballs. Taking a look at the diagram, given any node as a starting point, you can get to any other node. via a simple set of operations: firstChild, lastChild, nextSibling, previousSibling and parentNode.

Starting from html (our document element), I can get to h2 by saying documentElement - lastChild - lastChild - firstChild. From my h2 I can get to my h1 by saying parentNode - previousSibling - firstChild. The things and methods we can use to parse these documents are very robust and can be confusing at times, but to a large extent, this concept governs the entire api!

All together

Presentation Environment

Tree structures are efficiently and easily queried. Careful arrangement of nodes, careful use of attributes, and careful use of semantic elements impart a great deal of information about our content. Developers convert this structure into a meaningful presentation using a stylesheet language—the one we are becoming familiar with is css. The document structure can (and should) be the only spot where we store meaning. The presentation view of our application or website can (and should) be the only place where we manipulate the structure into a pleasing design.

Programming Environment

Programming languages, quite literally, know how to read, write, and manipulate everything regarding structured data. Many programming languages can work with the Document Object Model api, and the one concern ourselves with is, of course, JavaScript. The following list is copied verbatim from the Document Object Model Wikipedia Page

  • JavaScript can add, change, and remove all the HTML elements and attributes in the page
  • JavaScript can change all the CSS styles in the page
  • JavaScript can react to all existing events in the page
  • JavaScript can create new events in the page

Ponder: by this point you may be wondering, why would I need anything besides css?. The first thing you may notice is that you can’t add/remove content very robustly nor track events beyond 'hover'. More applicable to the dom, however, is an inability to work your way backwards up the tree (parentNode, previousSibling). Why might this be? Hint, think about what something like node - previousSibling - nextSibling is saying.

Review

Last week we went over classes and ID's, naming conventions and began an introduction to programming.

Semantic html

There are many important reasons to write your markup semantically, and while one reason may have more personal importance to you than another together they make a pretty compelling case as to why this is requisite, not just a good idea. Clear, lean code offers many advantages. It’s easy to fix, quick to download and causes fewer problems across different types of browsers. Relevant tags help screen readers, search engines and mobile devices interpret meaning in addition to layout.

Semantic structure is algorithmic

If not obvious by now, structuring a tree structure correctly means that a machine can make assumptions and parse a document efficiently. The obvious machine that comes to mind is a search engine.

Semantic structure is data

Sloppy or improper use of an html Element will beget redundant and inconsistent css application. Further, imprecise, arbitrary use of html is an imprecise, arbitrary dataset. Now, I suppose it is quite evident at this point (or will be soon) that JavaScript is a programming language. What may not be clear to a novice, is that programs are almost completely useless without a dataset. An imprecise arbitrary dataset is essentially nonsense, so for those of us who want to be programmers I want some extra thought put in to what this means.

Semantic structure is thoughtful

Something we think about far too little, especially on the path to becoming designers is this fact: one-fifth of our society has a disability. Whether a user is born with or has acquired a paralytic disorder, or a vision or cognitive impairment, or has some other technological barrier, nobody should be banned from participation because of corner-cutting. Far too often the design community forgets this, and it is nothing short of arrogant. Well-structured documents help assistive technologies parse and deliver content as intended.

Elements

A quick reminder, div and span have no inherent meaning. Many—professionals even—overuse them as a crutch. Remember, you're first goal is to add semantic value to your markup and to generate good data. If there is a better substitute for the tag you are using, use it. I will never penalize for using the wrong tag, but I will begin penalizing for usage of div/span when there is a better alternative. If you haven't made a habit of searching for or reviewing the Elements available, then you should start.

Media

Embedding Images

Images are very easy to work with, in fact, optimizing and saving images (say, in Photoshop) is actually difficult—and beyond the scope of this course. As such let's just revisit the basic embed of an image.

      <img src="valid/path.(jpg|png|gif|svg)" title="description" alt="alternative text (describe the image)"/>

Embedding Audio/video

Audio and video are only slightly more complicated. Depending on your format, you may have to supply multiple video types, but for the vast majority of users, an mp3 or m4a will work for audio or a mp4 for video.

      <audio src="valid/path.(mp3|m4a)" controls="controls">
        Your browser does not support the audio element.
      </audio>

      <video src="valid/path.(mp4|mov)" controls="controls" poster="valid/path..(jpg|png|gif|svg)">
        Your browser does not support the audio element.
      </video>

css combinators

We are familiar with descendant selectors, something like article section will allow us to style all any section so long as it has an article as an ancestor. Much of the time, this can be more than we want, but luckily we can limit the reach of the descendant chain with the use of a Child Selector. By simply changing our previous selector to article > section we only select sections that are direct children of an article. If you're not familiar with the difference between children and descendants yet, hit the books.

While not as immediately useful, adjacent sibling selectors can be utilized to apply classes based on if elements are directly next to each other. An example would be adding a top border all but the first element of a list with li + li { border-top: 1px solid #000; }. If say, you wanted all the paragraphs following the first one to have an indent, but couldn't be sure that it would be only p tags, you could switch to general sibling selectors and use a selector like p ~ p { text-indent: 1em; }.

Careful use of classes can simulate boolean behavior with CSS, and can be used to great effect—which is why I think css preprocessors (programs that write css for you) are somewhat unnecessary. Written way back in the W3C css 2.0 spec: CSS gives so much power to the class attribute, that authors could conceivably design their own document language based on elements with almost no associated presentation. Broken down, the rules that inform how you may apply classes are so powerful that we could introduce meta-level thinking (not you actually would, or should.)

With Javascript

Boolean Algebra, the namesake of George Boole, a 19th century mathematician who pioneered the system of logical operations, can take complicated sets of logic and break them into a fundamental binary solution that can only be one of two values: true and false. These Boolean values have straightforward properties, but as you combine them and use the algebraic systems, they quickly escalate in complexity.

true values and expressions
true (obviously), in JavaScript, this is written true
not false, !false
true or false, true || false
Five is greater than 2, 5 > 2
This name is the same 'Boabdil' == 'Boabdil'
This name is not the same 'Boabdil' != 'boabdil'
false values and expressions
false (obviously), in JavaScript, this is written as false
not true, !true
true and false, true && false

Order of Evaluation

Looking at the adjacent example, notice my careful use of parentheses. In general, always use parentheses for clarity, even when you fundamentally know the order of operations. Boolean operations are more obscure when it comes to order of evaluation, performing what is known as Short-circuit Evaluation, which is a type of optimization (and quite unnecessary for most JavaScript programming).

Fundamentally, without use of parentheses, the operation trys to find a true/false value as quickly as possible. Take the example and remove the parentheses and you'll see an example of this.

Coercion

While not particularly useful to a beginner programmer, type coercion is often a misunderstood and misused part of programming languages. In JavaScript, you have the following primitive types at your disposal: number, string, boolean, null and undefined. Trying to use any non-boolean type in a logical operation (any operation that uses a logical operator) may produce unexpected results as they are cast to booleans for comparison. Depending on the value they might be considered truthy or falsy. As you begin, try very hard not to rely on coercion until you understand it intuitively. In the meantime, here is a table to start getting familiar:

TRUTHY any non-zero number any non-empty string evaluating arrays and objects, even if empty (never do this!!)

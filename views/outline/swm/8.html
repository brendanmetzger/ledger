Case Study: A Lightbox and CSS

Css is needed to create an overlay. I'm going to show you one way, but there are many facets you need to begin to understand. Before we begin the discussion, I want you to investigate the following topics on your own: css positioning, z-index, overflow, the vh and vw units. Before we begin our work together you will highlight something you learned that you did not know before.

What is an overlay in principle
It's just a block level element, positioned in such a way that it overlays the entire page contents
It can cover the entire page, top to bottom, or just the visible screen 100vh and 100vw.
It will utilize position: absolute and some sort of z-index: (int)
The background can/should be locked
If the overlay fully covers page content, you don't need to worry about scrolling (although it might be annoying)
If the overlay is only covering the viewport, you will need to disable scrolling
The background itself
There are many CSS coloring techniques that support an alpha channel (providing opacity).
Understanding Hierarchy
The overlay layer is by definition a child of the body. Controlling it's visibility is something you need to spend some time thinking about, as there are several ways. This week, we'll focus exclusively on usage of classes, and dive in next week with some JavaScript.

Functions

The most important part of the way we write JavaScript is the notion of functions. Before we go down that road, we must understand a little bit about the namesake and the reasoning behind these methods.

In the beginning—and not particularly suprising considering the name—computers were primarily concerned with well, computing. Yes, giant calculators! I've tried introducing functions in very gentle ways, comparing them to cheeky things we know in real life; making pancakes and washing dishes, but that doesn't seem to convey the gravity of how important they are nor how to use them correctly. Most often you'll hear functions defined as a way make programs reusable, decreasing error and duplication and keeping your code organized.

While that definition is absolutely true, I'll argue it's only a nice byproduct of a nicely abstracted process, not process itself. In their purest form, functions take input, and return output, so let's start with that notion of purity and move on to pancakes and avoiding duplication. In this light, I'm going to attempt the opposite and talk about functions in the most terrifying way possible: mathematics.

We will spend the rest of class taking apart this monster and turning it into a function(s), all homemade, all using techniques outlined in 1715. Brains will melt, but you'll walk away knowing a few things: there is no such thing as magic, small details can lead to big ideas.

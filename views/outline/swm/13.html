Frameworks

Using a framework—from my experience—is only necessary in the following circumstance: you know what you want to make but don’t know how to do it, nor do you have time to learn how. Without decades of experience and bunch of free time, this is almost a guarantee.

The truth is that there is no shame in this, often we are just too proud to admit we don’t know everything (oddly, we have no trouble admitting we don’t have the time [though that’s often less true]). There are always problems that are difficult to solve and others with the time have gone to great length to solve already. Given plenty of experience and time, a framework is nothing more than short-cut; but without those two factors, they are often necessary.

That wasn't always the case. There was a point in the earlier days of web development where different browsers implemented both CSS and the DOM quite differently. Countless hours were spent fixing bugs and designing hacks so that one page would more or less function the same in different browsers. To save oursel or use a framework that fixed the bugs for you to give a consistent api. Long ago, in non-IE to bind an event listener you would use our familiar addEventListener, but in Internet Explorer, you would use addEvent. The spec agreed it should be addEventListener, but marketshare allowed some bigger players to do whatever they wanted.

So frameworks ironed this out for us—we didn’t have to write spaghetti code to grapple with inconsistencies. One such framework, say jQuery, might give us a new method called on that did all the event lister stuff for us,and more—it was a windfall at the time. Today, much of this is a non-issue, we can write nice clean, cross-browser code, but jQuery lives on because it has become very familiar.

And this is a caveat I'd mention now; when a framework prevents you from expanding your perspective, it is an awkward tool. Like a crutch that alleviates an ailment, it won’t provide an advantage over someone who can function without one. I was an early proponent and enthusiast of any tool that makes us better, mind, there is a large gulf between better and adequate when it comes to proficiency.

This is an important point I make: we should use frameworks often, and enjoy using them, but take to heart, in a best case world, you write your own code. Take a humble moment every now and then to remember that and let it push your design and your curiosity forward.

A review

The list of JS frameworks is long; I will boil down some interesting ones that I think have lasting value and indicate why.

Vanilla JS
A joke site used to talk about the state of the current language and that modern techniques available are actually more terse and straightforward, and efficient than the most popular frameworks.
jQuery
Everybody knows about this… with jQuery you go shopping for every plugin and interface widget ever made: light-boxes, carousels, parallax effects, oh my!
React
Lots of people are talking about this. That’s all I can say, as I will shamefully admit, I can’t help but find it incredibly intimidating and generally, overdoing it.
Snap SVG
A vector drawing framework (formerly called Raphaël). Scalable vector graphics are fun, but I daresay, a bit intimidating. This is a good toe-dip into that pool.
D3
A data-visualization library. Absolutely adored by data journalists, map geeks and statistics nerds.
Ramda
Just because… This is a functional library, that allows you to program in an entirely new style. It’s part genius, bewildering, sometimes ridiculous and entirely fascinating.

Some Thoughts on Code Sharing

I am always reevaluating what should and should not be presented to folks starting ot to learn so many new concepts in the web development domain, and typically it centers around, does this lead to good habits? That means I leave some good techniques (and some even easy ones), as well as many optimizations out on the curb. Optimizations are largely meaningless—consider an expensive racing bicycle—it may be half a kilogram lighter, but that is largely meaningless if you are not in excellent shape.

For this reason, I find it harder and harder to teach fundamentals when libraries like jQuery do so much optimization and you'll never know what is and is not relevant. As I researched this section for material, I stumbled on a well suited example.

Optimization

I have taught you at this point to always use the addEventListener('load', callback) in your script if you are going to query or modify the dom. In the jQuery framework, the authors say something similar, albeit by glossing over several details. My beef is twofold: first, they use a sloppy JavaScript technique to point out the native weakness and second, they negatively critique the load event as dangerous without stating how they have improved it.

I happen to know how they improved it; they simply used a more nuanced event, which would be addEventListener('DOMContentLoaded', callback), a very modest enhancement from what I've taught all along in this course. Now, are you ready for the caveats?

Cross-Browser Compatibility

The DOMContentLoaded event does not work in versions of internet explorer 8 and below, (a tiny tiny tiny market share mind you). A small enough marketshare that I would completely rule it out, or make a quick 3 second solution to solve this problem if necessary. Mind, 0.4% (or, 0.004 in decimal) use this browser globally. Far more people do not have functioning hands, mind you.

To a novice, that three second solution would likely not identify itself, so they might search for one, and find something called a polyfill. I found a polyfill for this very problem, which I read parsed and found to be questionable—particularly lines 172~173. I have scanned up and down, and though I understand the syntax, most of the technique goes way over my head. Here is the part I can explain: at line 172~173 it appears the author sometimes executes the callback twice (but doesn't really know and even admits that in the adjacent comment). Now it would be fair to ask—assuming I am right about a potential bug—why hasn’t it been identified? Because there is only a .004 chance that anyone will ever see it, and of that 0.004 that could, there is fraction of that percent that would even notice.

This is 400 lines of code and I'm telling you that with elegance and human ingenuity, it could be done with about 4. Remember that when you cut and paste, it might look like a genius work of engineering or it could be a bridge waiting to collapse under its own weight.

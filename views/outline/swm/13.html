Drawing

There are two fundamental ways to draw usin web technologies: using the canvas element and with Scalable Vector Graphics referred to as SVG. To draw with the canvas element, you start with an html element called canvas and from there, use a JavaScript API to make shapes, set fill colors, etc. SVG on the other hand, is and XML file, and these files can be constructed by vector drawing programs like Adobe Illustrator, embedded as an image like any other format (<img src="file.svg"/>) or embedded directly into an HTML page as an <svg> element

Using illustrator, we'll take a very quick tour over adding an svg to a document, and recall that we can now use JavaScript and CSS to adjust scenarios.

Frameworks

Using frameworks falls into some distinct patterns—I realize this may be insulting, but it’s really quite accurate.

  1. You don’t have time
  2. You don’t know how
  3. Probably Both

The truth is that there is no shame in this, often we are just too proud to admit we don’t know everything (oddly, we have no trouble admitting we don’t have the time [though that’s often less true]). There are always problems that are difficult to solve and others with the time have gone to great length to solve already. Given plenty of experience and time, frameworks are nothing more than convenient; but without those two factors, they can be necessary.

There was a point in JavaScript development where different browsers implemented the DOM differently—it was so ugly that you had either had to spend all your time fixing bugs or use a framework that fixed the bugs for you to give a consistent api. Long ago, in non-IE to bind an event listener you would use our familiar addEventListener, but in Internet Explorer, you would use addEvent. The spec agreed it should be addEventListener, but marketshare allowed some bigger players to do whatever they wanted.

So frameworks ironed this out for us—we didn't have to write spaghetti code to grapple with inconsistencies. One such framework, say jQuery, might give us a new method called on that did all the event lister stuff for us,and more—it was a windfall at the time. Today, much of this is a non-issue, we can write nice clean, cross-browser code, but jQuery lives on because it has become very familiar.

And this is a caveat I'd mention now; when a framework prevents you from expanding your perspective, it is an awkward tool. Like a crutch that alleviates an ailment, it won't provide an advantage over someone who can function without one. I was an early proponent and enthusiast of any tool that makes us better, mind, there is a large gulf between better and adequate when it comes to proficiency.

This is an important point I make: we should use frameworks often, and enjoy using them, but take to heart, in a best case world, you write your own code. Take a humble moment every now and then to remember that and let it push your design and your curiosity forward.

A review

The list of JS frameworks is long; I will boil down some interesting ones that I think have lasting value and indicate why.

Vanilla JS
A joke site used to talk about the state of the current language and that modern techniques available are actually more terse and straightforward, and efficient than the most popular frameworks.
jQuery
Everybody knows about this... with jQuery you go shopping for every plugin and interface widget ever made: light-boxes, carousels, parallax effects, oh my!
React
Lots of people are talking about this. That's all I can say, as I will shamefully admit, I can’t help but find it incredibly intimidating and generally, overdoing it.
Snap SVG
A vector drawing framework (formerly called Raphaël). Scalable vector graphics are fun, but I daresay, a bit intimidating. This is a good toe-dip into that pool.
D3
A data-visualization library. Absolutely adored by data journalists, map geeks and statistics nerds.
Ramda
Just because... This is a functional library, that allows you to program in an entirely new style. It's part genius, bewildering, sometimes ridiculous and entirely fascinating.
